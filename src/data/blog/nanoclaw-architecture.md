Telegram에서 "오늘 Kubernetes 뉴스 정리해줘"라고 메시지를 보낸다. 몇 초 후, AI가 웹을 검색하고, 기사를 읽고, 요약을 정리해서 답한다. 여기까지는 평범한 챗봇이다.

그런데 "Vault에 노트로 저장해줘"라고 추가하면, AI가 Obsidian vault에 마크다운 파일을 생성한다. "블로그에 발행해줘"라고 하면, git commit → git push → GitHub Actions 빌드까지 일어난다. Bash를 실행하고, 파일을 만들고, 외부 시스템과 상호작용하는 **풀스택 에이전트**다.

이 글은 그 메시지 한 줄이 어떤 경로를 거쳐 AI 에이전트가 되는지, 그리고 그 과정에서 **왜 이런 구조적 결정을 했는지**를 다룬다.

## 핵심 질문: AI에게 Bash를 줘도 되는가

AI 에이전트에게 터미널을 주면 강력해진다. 코드 실행, 파일 조작, 네트워크 요청 — 할 수 있는 일이 폭발적으로 늘어난다.

동시에 위험도 폭발한다. 에이전트가 `rm -rf /`를 실행하면? SSH 키를 읽어서 외부로 전송하면? 시스템 설정을 바꿔버리면?

NanoClaw의 답은 명확하다. **줘도 된다. 단, 격리된 공간에서만.**

Claude Agent SDK(= Claude Code)는 원래 개발자의 로컬 터미널에서 동작하도록 만들어졌다. NanoClaw는 이것을 Docker 컨테이너 안에 가둬서, Telegram이라는 원격 인터페이스를 통해 안전하게 사용할 수 있게 만든 프로젝트다.

## 두 세계의 분리: Host와 Container

NanoClaw의 아키텍처를 한 문장으로 요약하면 이렇다:

> **Host는 배달부, Container는 두뇌.**

```
사용자 (Telegram)
  │
  ▼
┌─────────────────────────────────────┐
│ Host (Node.js) — 오케스트레이터     │
│                                     │
│  Telegram Bot (grammy)              │
│  메시지 라우팅                       │
│  SQLite DB                          │
│  스케줄러 (cron)                     │
│  IPC 감시                           │
│                                     │
│  ※ AI 처리를 하지 않는다             │
└──────────┬──────────────────────────┘
           │ spawn() + stdin JSON
           ▼
┌─────────────────────────────────────┐
│ Docker Container — 동적 생성/종료    │
│                                     │
│  Claude Agent SDK                   │
│  Bash · 파일 · 웹 브라우징            │
│  MCP · 스킬 시스템                   │
│                                     │
│  ※ 모든 AI 작업이 여기서 일어난다     │
└─────────────────────────────────────┘
           │ stdout JSON 스트리밍
           ▼
사용자 (Telegram)
```

Host는 Telegram 메시지를 받아서 컨테이너에 전달하고, 결과를 받아서 Telegram에 보낸다. 그게 전부다. AI가 뭘 하든 — 웹을 검색하든, 파일을 쓰든, `apt install`을 하든 — 컨테이너 안에서만 일어난다.

이 분리가 만드는 보장:

- **에이전트가 Bash를 마음껏 써도 Host 시스템은 무사하다**
- 에이전트가 죽어도(OOM, 무한루프) Host는 정상 동작한다
- 그룹 A의 에이전트가 그룹 B의 데이터에 접근할 수 없다
- API 키와 OAuth 토큰이 컨테이너 파일시스템에 남지 않는다

## 메시지 한 줄의 여정

Telegram에서 "Vault에 Kubernetes 정리 노트 만들어줘"를 보내면 벌어지는 일을 따라가 보자.

**1단계: Telegram → Host**

grammy 라이브러리가 Telegram의 `getUpdates` API를 Long Polling으로 계속 호출하고 있다. 새 메시지가 감지되면 오케스트레이터가 받는다.

**2단계: Host → Container 생성**

오케스트레이터가 해당 그룹의 설정을 로드하고, `docker run -i` 명령으로 새 컨테이너를 생성한다. 이때 그룹에 맞는 볼륨이 마운트된다. main 그룹이면 Vault와 Portfolio 레포가 마운트되고, 일반 그룹이면 자기 폴더만 보인다.

**3단계: stdin으로 프롬프트 전달**

프롬프트와 함께 Claude API 인증에 필요한 OAuth 토큰이 stdin으로 전달된다. 환경변수도 아니고, 파일도 아니고, 프로세스의 표준 입력이다. 컨테이너가 종료되면 흔적이 남지 않는다.

**4단계: Claude가 작업 수행**

컨테이너 안에서 Claude Agent SDK가 Claude API를 호출한다. Claude가 "Vault에 노트를 만들려면 파일을 생성해야겠다"고 판단하면, Bash로 파일을 쓴다. 웹 검색이 필요하면 WebFetch를 호출한다. 모든 도구 사용이 컨테이너 안에서 일어난다.

**5단계: stdout으로 결과 반환**

결과가 stdout으로 스트리밍된다. 출력의 시작과 끝에 마커 문자열이 삽입되어 있어서, Host가 이를 감지해 JSON을 파싱한다.

**6단계: 컨테이너 종료**

응답이 끝나면 컨테이너가 종료된다. 다음 메시지가 오면 새 컨테이너가 생성된다. 매번 깨끗한 상태에서 시작하는 ephemeral 패턴이다.

전체 과정이 수 초에서 수십 초. 사용자 입장에서는 Telegram에서 대화하는 것처럼 느껴진다.

## 왜 HTTP가 아니라 stdin/stdout인가

Host와 Container가 통신하는 방식이 독특하다. REST API 서버를 띄우는 게 아니라, 프로세스의 stdin/stdout 파이프를 사용한다.

```
Host                           Container
  │                               │
  ├── spawn("docker run -i ...") ─┤
  │                               │
  ├── stdin.write(JSON) ─────────►│  프롬프트 + OAuth 토큰
  │                               │
  │◄──────────── stdout 스트리밍 ──┤  결과 JSON
  │                               │
```

처음에는 이상하게 느껴질 수 있다. HTTP가 훨씬 익숙하고 범용적이니까. 하지만 이 맥락에서는 파이프가 정답이다:

1. **포트를 열지 않는다**: 네트워크 리스닝이 없으므로 외부에서 접근할 수 없다
2. **인프라가 없다**: 서버 구동, 포트 관리, 연결 풀링이 필요 없다
3. **시크릿이 안전하다**: OAuth 토큰을 stdin으로 보내고 끝. 환경변수나 파일에 남지 않는다

Host의 Node.js 프로세스가 `child_process.spawn()`으로 Docker를 실행하고, 커널의 fork() + exec()가 파이프를 만든다. 운영체제가 제공하는 가장 기본적인 IPC 메커니즘이다.

## 공인 IP 없이 동작하는 이유

AI 어시스턴트를 서버에 올리는 게 일반적이다. 하지만 NanoClaw의 철학은 다르다: **개인 노트북에서 돌아간다.**

집 WiFi, 카페, 테더링 — 어디서든 인터넷만 있으면 동작해야 한다. 이걸 가능하게 하는 게 Telegram의 Long Polling이다.

| 방식 | 동작 | 필요한 것 |
|------|------|-----------|
| **Webhook** | Telegram이 내 서버로 POST | 공인 IP, 도메인, HTTPS |
| **Polling** | 내가 Telegram에 GET 반복 | 인터넷 연결 |

Webhook은 Telegram이 나를 찾아와야 하니까 주소가 필요하다. Polling은 내가 Telegram을 찾아가니까 주소가 필요 없다. NAT 뒤에 있어도, IP가 바뀌어도, outbound 연결만 되면 즉시 동작한다.

WiFi에서 테더링으로 전환해도? IP가 바뀌지만 새 outbound 연결을 맺으면 그만이다. 끊김 없이 계속 동작한다.

## 보안: 6개 레이어

에이전트에게 Bash를 주면서 안전하려면, 한 겹이 아니라 여러 겹의 방어가 필요하다.

**1. 파일시스템 격리**

그룹별로 Docker 볼륨 마운트가 다르다. A 그룹의 컨테이너에서는 B 그룹의 폴더가 아예 보이지 않는다. 마운트 자체가 안 되니까 우회할 방법이 없다.

**2. 프로젝트 보호**

NanoClaw 소스코드가 있는 프로젝트 루트는 read-only로 마운트된다. 관리자 권한이 있는 main 그룹조차 프로젝트 코드를 직접 수정할 수 없다.

**3. 시크릿 관리**

Claude API 인증에 필요한 OAuth 토큰은 stdin으로만 전달된다. 컨테이너 안의 환경변수, 파일시스템, 프로세스 목록 어디에도 토큰이 남지 않는다. `docker inspect`로 봐도 안 나온다.

**4. 마운트 허용 목록**

`mount-allowlist.json`에 허용된 경로만 마운트할 수 있다. 누군가 설정을 바꿔서 `/etc/passwd`를 마운트하려 해도 허용 목록에 없으면 거부된다.

**5. IPC 격리**

Host-Container 간 파일 기반 IPC도 그룹별로 별도 디렉토리를 사용한다. A 그룹의 IPC 명령이 B 그룹에 영향을 줄 수 없다.

**6. 세션 격리**

Claude Agent SDK의 세션 데이터(`.claude/` 디렉토리)도 그룹별로 분리된다. 대화 기록, 설정, 캐시가 섞이지 않는다.

## 스킬: 같은 시스템, 다른 맥락

NanoClaw는 Claude Code 위에 만들어졌다. 그래서 Claude Code의 스킬 시스템을 그대로 활용하는데, 재미있는 점은 **같은 스킬 시스템이 두 가지 전혀 다른 맥락에서 동작한다**는 것이다.

### 개발자가 쓰는 스킬 (Host)

개발자가 터미널에서 `claude`를 실행하고 사용하는 스킬이다. NanoClaw 자체를 구성하고 확장하는 "메타 스킬"이다.

`/add-telegram`을 실행하면 Telegram 채널 관련 코드가 프로젝트에 자동으로 추가된다. `/setup`은 최초 설치와 인증을 처리하고, `/update-nanoclaw`는 upstream 변경사항을 동기화한다. NanoClaw를 코딩 없이 확장할 수 있게 하는 코드 생성 도구다.

### 에이전트가 쓰는 스킬 (Container)

AI 에이전트가 사용자 메시지에 응답하면서 사용하는 도구다. `container/skills/`에 정의하면 컨테이너 실행 시 자동으로 복사된다.

`agent-browser`는 웹 페이지를 열고, 클릭하고, 폼을 작성하고, 스크린샷을 찍는다. `vault-to-blog`는 Obsidian 노트를 블로그 포맷으로 변환하고 git push까지 한다. 에이전트의 능력을 스킬 파일 하나로 확장할 수 있다.

같은 스킬 시스템인데, 하나는 NanoClaw를 만드는 데 쓰이고, 하나는 NanoClaw가 일하는 데 쓰인다. Claude Code의 스킬 아키텍처가 이런 이중 활용에 자연스럽게 맞아떨어졌다.

## 멀티 테넌시: 하나의 인스턴스, 여러 그룹

하나의 NanoClaw로 여러 Telegram 그룹과 DM을 서빙할 수 있다. 각 그룹은 서로 다른 권한과 격리 수준을 가진다.

| 구분 | 트리거 | 권한 |
|------|--------|------|
| **main** | 모든 메시지 처리 | 프로젝트 read-only, 그룹 관리, 전체 태스크 조회 |
| **일반 그룹** | @멘션 필요 | 자기 그룹 폴더만 접근 |
| **DM** | 모든 메시지 처리 | 자기 그룹 폴더만 접근 |

main은 관리자 채널이다. 다른 그룹을 등록/해제하고, 예약 작업을 관리하고, 프로젝트 소스를 (읽기 전용으로) 확인할 수 있다. 일반 그룹은 자기 폴더만 보이고, @멘션이 있어야 반응한다.

격리가 볼륨 마운트 레벨이라는 점이 중요하다. 소프트웨어적 접근 제어가 아니라, 컨테이너에 마운트 자체를 안 하니까 우회할 방법이 없다.

## 왜 이 구조를 선택했는가

AI 에이전트 아키텍처를 설계할 때 세 가지 선택지가 있었다:

**프로세스 내 직접 실행** — 가장 간단하다. Host의 Node.js 프로세스에서 Claude API를 직접 호출하면 된다. 하지만 에이전트가 Bash를 실행하면 Host 시스템에서 그대로 실행된다. 격리가 없다.

**VM 기반 격리** — 가장 강력하다. 커널 수준의 완전한 격리를 제공한다. 하지만 VM은 무겁다. 메시지 하나마다 VM을 띄우면 시작 시간이 수십 초다. 개인 노트북에서 돌리기엔 리소스도 부담이다.

**컨테이너 기반 격리** — NanoClaw가 선택한 방식이다. Docker 컨테이너는 수 초 만에 시작되고, 볼륨 마운트로 유연하게 파일시스템을 제어할 수 있고, 리소스 오버헤드가 적다. 엔터프라이즈급 보안은 아니지만, "실수로 Host를 망가뜨리는" 것은 확실히 막는다.

그리고 Claude Agent SDK 위에 만들어졌다는 점이 결정적이었다. 스킬 시스템, MCP 도구, 에이전트 기능을 처음부터 만들 필요가 없었다. Claude Code가 터미널에서 할 수 있는 모든 것을, Docker 컨테이너 안에서 그대로 할 수 있게 감싸는 것이 NanoClaw의 본질이다.

## 서비스 운영

macOS의 launchd로 관리된다.

| 서비스 | 역할 | 실행 방식 |
|--------|------|-----------|
| 오케스트레이터 | 메시지 수신/발신, 라우팅 | launchd 상주 프로세스 |
| Cloudflare Tunnel | 외부 서비스 연결 | launchd 상주 프로세스 |
| Container (에이전트) | AI 작업 수행 | 메시지마다 동적 생성/종료 |

컨테이너가 상주하지 않는다는 점이 핵심이다. 메시지가 올 때 생성되고, 응답이 끝나면 종료된다. 이 ephemeral 패턴은 두 가지 이점을 준다:

1. **리소스 절약**: 대기 중일 때 Docker 컨테이너가 메모리를 차지하지 않는다
2. **깨끗한 상태**: 매번 새 컨테이너에서 시작하므로 이전 세션의 부작용이 없다

## 정리

NanoClaw의 아키텍처는 하나의 질문에서 시작되었다: **AI 에이전트에게 Bash를 줘도 안전한 구조가 가능한가?**

Host-Container 분리, stdin/stdout 파이프 통신, 그룹별 볼륨 마운트 격리, ephemeral 컨테이너 패턴. 이 네 가지 설계 결정이 그 질문에 대한 답이다.

결과적으로 Telegram에서 메시지를 보내면, 격리된 Docker 컨테이너 안에서 Claude가 코드를 실행하고, 파일을 만들고, 웹을 탐색하고, git push까지 한다. Host 시스템은 안전한 채로. 그리고 이 모든 것이 Claude Agent SDK 위에서 — 바퀴를 다시 발명하지 않고 — 동작한다.
